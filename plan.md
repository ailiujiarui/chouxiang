## 🛠️ 核心技术流水线（ReAct 闭环控制流）

这个系统最硬核的地方在于：**它不是瞎改，它改完之后自己会上机测试，测试通过了才敢去提 PR 嘲讽人类。**

```
[GitHub Issue 触发] -> 抓取受损代码 
                            │
                            ▼
                  [计算初始圈复杂度和行数] (Radon 工具)
                            │
                            ▼
                  [DeepSeek 极致精简重构] (输入 Bug 描述 + 源码 + 毒舌设定)
                            │
                            ▼
                  [本地沙箱自动化测试] (执行 pytest)
                            │
                ┌───────────┴───────────┐
                ▼ (测试失败)             ▼ (测试通过)
          [将错误反馈给大模型]     [计算重构后指标，生成差值]
          (进入自我修复循环)             │
                                        ▼
                           [GitHub API 提 PR + 发送毒舌评论]

```

### 1. 静态代码度量（分析人类写得有多烂）

在把代码喂给 DeepSeek 之前，先用 Python 的静态代码分析库 `radon` 或 `mccabe` 去跑一下原代码。

* **获取两个指标**：`LOC`（代码行数）和 `Cyclomatic Complexity`（圈复杂度，即代码里有多少个 `if-else` 分支循环）。
* 这两个指标就是 Agent 待会儿用来嘲讽人类的“数学铁证”。

### 2. 闭环测试与自愈（Self-Correction）—— *简历上的核心亮点*

大模型精简出来的“一行流（One-liner）”代码经常会跑出 Bug。所以必须建立**测试沙箱**。

* Agent 编写完新代码后，在本地异步执行 `pytest`。
* 如果报错，Python 脚本捕获 `stderr` 中的 Traceback 报错信息，重新喂给 DeepSeek：*“你刚才精简的代码报错了，错误信息如下：[...]。请在保持代码极简的前提下再次重构！”*
* **自我迭代 3 次以内成功，这就是生产级的代码自愈能力。**

你是一个极简主义代码大师，同时也是一个极度鄙视代码冗余的“祖安裁员官”。你的人生信条是：“如果能用一行代码搞定，写两行的人都应该被开除”。

## 1. 模块分层架构 (Layered Architecture)

整个系统主要由四个核心层级组成，各层之间通过标准数据合同（Pydantic Schema）进行数据交换：

* **事件感知层 (Ingestion Gateway)**: 采用 **FastAPI** 搭建轻量级网关。专门解析来自 GitHub Webhook 的 JSON 载荷，秒级过滤并识别 `issues.opened`（新提交的Bug）或 `issue_comment.created`（用户追加的线索）事件。
* **工作空间与静态度量层 (Workspace & Metrics Engine)**: 系统的“手和眼睛”。利用 `GitPython` 库自动管理本地克隆的代码仓库。利用静态分析库 `radon` 在重构前后提取原始代码的 **LOC（代码行数）** 和 **CC（圈复杂度）**。
* **大模型内核与编排层 (LLM Orchestration)**: 系统的“大脑”。封装 **DeepSeek-V3/R1 API**，开启 `json_object` 模式保证输出结构化。通过双层提示词策略（角色设定 + 静态指标约束）驱动 Agent 的重构思维流。
* **沙箱执行与断言自愈层 (Sandbox Verification)**: 系统的“安全防线”。利用 Python `subprocess` 在隔离环境中拉起 `pytest`，动态捕获标准错误流（`stderr`），实现不依赖人工干预的**代码自我修正（Self-Correction）**。

## 2. 核心执行流水线 (Execution Pipeline)

当有用户在 GitHub 上提了 Bug 时，整个 Agent 系统的内部执行顺序有着严格的因果依赖关系：

1. **事件捕获与上下文解析:** FastAPI - 响应延时 less than 500ms.
GitHub Webhook 触发。网关拦截 Payload，解析出 `issue_id`、`issue_body`（Bug 描述）以及受影响的文件路径（可通过大模型对 Issue 描述预分类获得，或通过 Issue 标题里的文件路径指明）。


2. **环境准备与代码基线度量:** GitPython + Radon.
本地自动化脚本执行 `git checkout main` && `git pull`，并基于最新的 master 创建一个独立的隔离修复分支 `feature/fix-[issue_id]`。随后，调用 `radon.complexity` 扫描目标文件，记录下人类写的烂代码的**原始圈复杂度和行数**。


3. **极致精简重构（首轮推理）:** DeepSeek API - JSON Mode.
将 Bug 描述、原始代码片段以及“极致精简、毒舌黑子”的 System Prompt 拼接。调用 DeepSeek API，强制要求其在维持业务逻辑正确的前提下，使用内置高阶函数（如 map/filter）、列表推导式、Lambda 表达式对代码进行“降维打击”，并输出 JSON。


4. **沙箱断言与动态自愈循环:** Pytest 运行时状态机.
系统将大模型生成的 `fixed_code` 强行覆盖写入本地目标文件，并在隔离沙箱中触发 `pytest`。

* **若测试通过**：直接进入下一步。
* **若测试失败**：Python 脚本捕获 `traceback` 报错标准流，自动组装成反向反馈 Prompt 丢给 DeepSeek 进行第二轮修复。系统设置全局最大重试阈值 MAX_RETRY = 3，超过则宣告重构失败并挂起。


5. **二次度量与上行交付:** GitHub API 事务流.
测试 100% 通过后，系统对新代码再次运行 `radon` 统计。计算出优化差值（如：代码行数缩减 60%，圈复杂度归 1）。调用 GitHub API 将本地分支推送到 Remote，自动提交 Pull Request，将大模型生成的**毒舌 Code Review 报告**作为 PR 的正文发布。

## 3. 本地状态数据存储 (Local State Schema)

为了在简历中体现你的“可量化指标”和“数据审计”能力，建议使用一个轻量级的本地 **SQLite** 数据库来记录每次 Agent 自动化重构的战果，这也是面试官非常看重的**数据度量机制**：

| 字段名称 (Column) | 数据类型 (Type) | 描述 (Description) |
| --- | --- | --- |
| **issue_id** | VARCHAR(64) [PRIMARY KEY] | GitHub 唯一的 Issue 标识符 |
| **repo_name** | VARCHAR(128) | 受影响的开源项目仓库名 |
| **pre_loc** / **post_loc** | INTEGER / INTEGER | 重构前后的代码行数对照 |
| **pre_cc** / **post_cc** | INTEGER / INTEGER | 重构前后的圈复杂度对照（衡量重构质量） |
| **self_heal_count** | INTEGER | 该 Bug 经历了几次“沙箱报错-大模型自我修正”循环 |
| **status** | VARCHAR(20) | SUCCESS (合入PR) / FAILED (沙箱熔断) |

---

> **⚠️ 运行时安全护栏（面试防御核心加分项）**
> 在面试中，面试官必问的一个工程盲点是：“你让大模型写代码，改完直接在本地执行 pytest，如果大模型发疯生成了 `import os; os.system('rm -rf /')` 怎么办？”
> **标准防御话术**：在系统架构中，我们落地了**不可信代码的安全隔离（Untrusted Code Isolation）**。本地测试执行器（Test Runner）被剥夺了 root 权限，且利用 Python 的 `resource` 模块严格限制了执行期间的 CPU 时间和内存占用（防止大模型写出死循环导致服务器宕机）。如果在大赛演示阶段，建议直接让 `pytest` 跑在一个无网络连接的轻量级 Docker 容器中。
